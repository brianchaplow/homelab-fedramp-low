# au-12 - \[Audit and Accountability\] Audit Record Generation

## Control Statement

- \[a.\] Provide audit record generation capability for the event types the system is capable of auditing as defined in [AU-2a](#au-2_smt.a) on [system components];

- \[b.\] Allow [personnel or roles] to select the event types that are to be logged by specific components of the system; and

- \[c.\] Generate audit records for the event types defined in [AU-2c](#au-2_smt.c) that include the audit record content defined in [AU-3](#au-3).

## Control Assessment Objective

- \[AU-12a.\] audit record generation capability for the event types the system is capable of auditing (defined in AU-02_ODP[01]) is provided by [system components];

- \[AU-12b.\] [personnel or roles] is/are allowed to select the event types that are to be logged by specific components of the system;

- \[AU-12c.\] audit records for the event types defined in AU-02_ODP[02] that include the audit record content defined in AU-03 are generated.

## Control guidance

Audit records can be generated from many different system components. The event types specified in [AU-2d](#au-2_smt.d) are the event types for which audit logs are to be generated and are a subset of all event types for which the system can generate audit records.

### This System

Audit record generation is implemented through a dual-stack architecture. The primary stack runs on brisket: Wazuh Manager 4.14.4 generates audit records for all 15 in-boundary agents (FIM via syscheck, process monitoring via syscollector, log analysis rules, SCA policy checks, and vulnerability detection) and ships them to `wazuh-alerts-4.x-*` on the Wazuh Indexer. Suricata on smokehouse eth4 generates IDS alert records (eve.json) shipped via the smokehouse Wazuh agent into the same index family. OPNsense firewall ships syslog (UDP 514) directly to Wazuh Manager, generating audit records for firewall allow/deny events. The secondary stack runs on haccp: Zeek 8.1.1 on span0 generates protocol metadata records across all monitored log types (conn, dns, http, ssl, ssh, files, x509, notice); Filebeat ships them to Logstash:5044 where the 5-stage zeek-enrichment pipeline generates enriched records with OpenCTI TI fields, novel-entity classification, and Ollama LLM verdict into `logs-zeek.haccp-default-*` on ELK Elasticsearch. Arkime on haccp generates PCAP index records in its own database, keyed to community-id for cross-stack correlation.

AU-12(b) selection authority: Brian Chaplow as sole operator selects event types to be logged by specific components through three interfaces -- Wazuh Dashboard (brisket:5601) for rule group enable/disable and agent-level policy overrides, Wazuh Manager ossec.conf and rules XML for persistent changes, and the Logstash `zeek-enrichment.conf` pipeline configuration on haccp for Zeek-tier event routing and enrichment selection. The `oscal/component-definition.json` enumerates all 7 in-boundary components with audit capability mapped, providing the OSCAL-native artifact confirming that AU-12(a) coverage spans all components where audit capability is deployed. MokerLink (the L3 switch) does not currently ship logs to any collection point -- OPNsense firewall logs substitute for boundary traffic events at the VLAN gateway layer.
