# ac-18 - \[Access Control\] Wireless Access

## Control Statement

- \[a.\] Establish configuration requirements, connection requirements, and implementation guidance for each type of wireless access; and

- \[b.\] Authorize each type of wireless access to the system prior to allowing such connections.

## Control Assessment Objective

- \[AC-18a.\]

  - \[AC-18a.[01]\] configuration requirements are established for each type of wireless access;
  - \[AC-18a.[02]\] connection requirements are established for each type of wireless access;
  - \[AC-18a.[03]\] implementation guidance is established for each type of wireless access;

- \[AC-18b.\] each type of wireless access to the system is authorized prior to allowing such connections.

## Control guidance

Wireless technologies include microwave, packet radio (ultra-high frequency or very high frequency), 802.11x, and Bluetooth. Wireless networks use authentication protocols that provide authenticator protection and mutual authentication.

### This System

AC-18 is not applicable to this system. No in-boundary MSS components are connected via wireless. All in-boundary hosts -- brisket (ThinkStation), haccp (ThinkStation P340 Tiny), smokehouse (QNAP TVS-871), dojo VM (on pitcrew), regscale VM (on smoker), OPNsense (Protectli VP2420), and MokerLink -- communicate exclusively over wired Ethernet connections terminated at the MokerLink 10G managed switch, as documented in the 2026-04-07 rack build (`/c/Projects/reference/network.md` and `rack_build_2026-04-07.md`). The `inventory/overlay.yaml` hardware model entries for all in-boundary assets describe wired or Proxmox-bridged network connectivity with no WiFi adapters. The ASUS router (family LAN 192.168.50.0/24 and family DMZ 192.168.100.0/24) is out-of-boundary; VLAN 50 (IoT) is WiFi-served but contains no MSS components. Cross-reference AC-19 (mobile devices, also not applicable) and AC-17 (the documented remote access path, which uses Tailscale over the wired infrastructure connection).
