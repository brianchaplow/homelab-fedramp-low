# cm-11 - \[Configuration Management\] User-installed Software

## Control Statement

- \[a.\] Establish [policies] governing the installation of software by users;

- \[b.\] Enforce software installation policies through the following methods: [methods] ; and

- \[c.\] Monitor policy compliance [frequency].

## Control Assessment Objective

- \[CM-11a.\] [policies] governing the installation of software by users are established;

- \[CM-11b.\] software installation policies are enforced through [methods];

- \[CM-11c.\] compliance with [policies] is monitored [frequency].

## Control guidance

If provided the necessary privileges, users can install software in organizational systems. To maintain control over the software installed, organizations identify permitted and prohibited actions regarding software installation. Permitted software installations include updates and security patches to existing software and downloading new applications from organization-approved "app stores." Prohibited software installations include software with unknown or suspect pedigrees or software that organizations consider potentially malicious. Policies selected for governing user-installed software are organization-developed or provided by some external entity. Policy enforcement methods can include procedural methods and automated methods.

### This System

The system is a single-operator homelab SOC with no general-purpose user accounts on in-boundary hosts beyond the operator (Brian Chaplow). The operative software installation policy is operator-authored: all new software installations must be documented in CLAUDE.md §Service Inventory and committed to the homelab-fedramp-low or homelab-soc-portfolio git repository before the service is considered part of the approved baseline. Docker is the required deployment mechanism for new services on brisket and GRC VMs, providing container namespace isolation and explicit port binding as a technical enforcement layer. Ubuntu 24.04 VMs (dojo, regscale) use `unattended-upgrades` for security patches only -- no user-initiated package installation outside operator SSH sessions is expected or authorized (ADR 0002 §Infrastructure state). SSH key-only authentication to all in-boundary hosts is the primary logical access control that restricts software installation to the authorized operator.

UFW default-deny on dojo and regscale prevents inbound lateral software installation attempts. Wazuh syscollector captures the installed package set on all enrolled agents at enrollment and on a polling cadence, providing change detection if unauthorized packages appear on any in-boundary host; these snapshots feed the monthly ConMon pipeline (`pipelines/ingest/inventory.py`). Wazuh SCA checks validate OS-level hardening against unauthorized software indicators. The single-operator model significantly reduces the risk profile of user-installed software -- there is no general user population to govern, only the operator themselves. The primary gaps driving `partial` status are: (1) the policy is informal (CLAUDE.md convention, not a standalone policy document); and (2) Wazuh syscollector detects unauthorized software changes but does not block them at installation time.

#### Implementation Status: partial
