# sc-13 - \[System and Communications Protection\] Cryptographic Protection

## Control Statement

- \[a.\] Determine the [cryptographic uses] ; and

- \[b.\] Implement the following types of cryptography required for each specified cryptographic use: [types of cryptography].

## Control Assessment Objective

- \[SC-13a.\] [cryptographic uses] are identified;

- \[SC-13b.\] [types of cryptography] for each specified cryptographic use (defined in SC-13_ODP[01]) are implemented.

## Control guidance

Cryptography can be employed to support a variety of security solutions, including the protection of classified information and controlled unclassified information, the provision and implementation of digital signatures, and the enforcement of information separation when authorized individuals have the necessary clearances but lack the necessary formal access approvals. Cryptography can also be used to support random number and hash generation. Generally applicable cryptographic standards include FIPS-validated cryptography and NSA-approved cryptography. For example, organizations that need to protect classified information may specify the use of NSA-approved cryptography. Organizations that need to provision and implement digital signatures may specify the use of FIPS-validated cryptography. Cryptography is implemented in accordance with applicable laws, executive orders, directives, regulations, policies, standards, and guidelines.

### This System

Four cryptographic uses are identified across the MSS boundary and implemented with the following cryptographic types. HTTPS/TLS (TLS 1.2/1.3 via OpenSSL) is used for Wazuh Manager API (brisket:55000), Wazuh Indexer/OpenSearch (brisket:9200), ELK Elasticsearch (haccp:9200), and Wazuh Dashboard (brisket:5601), providing ECDHE key exchange, AES-GCM bulk encryption, and HMAC or AEAD integrity protection. WireGuard via Tailscale (ChaCha20-Poly1305 authenticated encryption) protects all remote administrative access from PITBOSS to in-boundary hosts. OpenSSH (AES-256-GCM over ECDH key exchange, or Ed25519 host keys) provides cryptographic protection for host management sessions and PCAP archival rsync from haccp to smokehouse. Wazuh enrollment key exchange (Wazuh-native symmetric key over TLS) secures agent enrollment on brisket:1515. All implementations use standard open-source primitives -- OpenSSL, the WireGuard kernel module, and OpenSSH -- running on standard Ubuntu LTS kernels.

Gap: no FIPS 140-2 or FIPS 140-3 validated cryptographic modules are in use. This is a homelab environment running standard Linux kernel cryptography rather than FIPS-enabled builds. All algorithms in active use (AES, SHA-256, ECDHE, ChaCha20-Poly1305, Ed25519) are NIST-approved, satisfying the algorithm coverage requirement even without formal module validation. For a production FedRAMP authorization this would require FIPS-enabled OpenSSL builds and validated kernel crypto modules -- that hardening path is deferred as a future phase item. Status is partial for FIPS validation gap; all identified cryptographic uses have appropriate algorithm implementations deployed.

#### Implementation Status: partial
