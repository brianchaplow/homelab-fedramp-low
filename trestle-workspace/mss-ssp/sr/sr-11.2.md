# sr-11.2 - \[Supply Chain Risk Management\] Configuration Control for Component Service and Repair

## Control Statement

Maintain configuration control over the following system components awaiting service or repair and serviced or repaired components awaiting return to service: [system components].

## Control Assessment Objective

- \[SR-11(02)[01]\] configuration control over [system components] awaiting service or repair is maintained;

- \[SR-11(02)[02]\] configuration control over serviced or repaired [system components] awaiting return to service is maintained.

### This System

Configuration control over components awaiting or returning from service is maintained through the hardware inventory baseline in `inventory/overlay.yaml` (model, function, asset tag for all in-boundary hosts) and the Wazuh syscollector continuous hardware inventory (CPU, RAM, NIC MACs on all in-boundary agents). The syscollector data at `https://10.10.20.30:55000/syscollector/{agent_id}/hardware` provides the pre-service and post-service hardware state comparison baseline -- unexpected changes (MAC drift, RAM reduction) would surface in delta queries.

For in-boundary VMs (dojo VMID 201, regscale VMID 301), the known-good VM configurations are committed to the repository at `deploy/proxmox/dojo-vm-config.yaml` and `deploy/proxmox/regscale-vm-config.yaml`. Any service, repair, or restore of these VMs is validated against these configuration files post-operation, as documented in ADR 0002.

The 2026-04-07 rack consolidation drive swaps are documented in CLAUDE.md rack consolidation notes (haccp 2TB Samsung 990 EVO Plus PCAP drive, pitcrew +512GB, smoker +1TB). The post-swap configuration state is reflected in `inventory/overlay.yaml`. However, no formal pre-swap to post-swap configuration comparison document or maintenance sign-off record was created. This gap -- no formal tagging of components as "awaiting service" and no written return-to-service verification step -- is acknowledged and accepted at the partial rating.
