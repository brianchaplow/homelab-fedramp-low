# ia-7 - \[Identification and Authentication\] Cryptographic Module Authentication

## Control Statement

Implement mechanisms for authentication to a cryptographic module that meet the requirements of applicable laws, executive orders, directives, policies, regulations, standards, and guidelines for such authentication.

## Control Assessment Objective

mechanisms for authentication to a cryptographic module are implemented that meet the requirements of applicable laws, executive orders, directives, policies, regulations, standards, and guidelines for such authentication.

## Control guidance

Authentication mechanisms may be required within a cryptographic module to authenticate an operator accessing the module and to verify that the operator is authorized to assume the requested role and perform services within that role.

### This System

Cryptographic module authentication in the Managed SOC Service relies on platform-provided cryptographic libraries and standard authentication mechanisms throughout.

On all in-boundary Linux hosts (brisket, haccp, dojo, regscale on Ubuntu 24.04; smokehouse, smoker, pitcrew, sear on their respective OS versions), OpenSSH authenticates access to the host operating system using the host kernel's built-in cryptographic primitives via libcrypto. Ubuntu ships OpenSSH linked against OpenSSL 3.x, which carries FIPS 140-3 CMVP validation certificates. The homelab does not operate in FIPS-enforced mode (a documented posture limitation noted below), but the underlying cryptographic library meets CMVP standards. TLS sessions for HTTPS services -- Wazuh Dashboard, Shuffle SOAR, and Velociraptor -- use OpenSSL-provided TLS 1.2 or TLS 1.3 with standard cipher suites; the cryptographic module (OpenSSL) is authenticated via APT's GPG-signed package manifests at install time. The Wazuh single-node Docker stack uses autogenerated TLS certificates for inter-component communication between the indexer, manager, and dashboard, with certificate verification handled by the OpenSearch security plugin. Tailscale WireGuard tunnels use Curve25519 and ChaCha20-Poly1305, implemented in the Linux kernel module, validated by upstream kernel maintainers. No proprietary or custom cryptographic modules are in use.

A known posture limitation: FIPS 140-3 validated mode is not enforced on any in-boundary host. A production FedRAMP deployment would require FIPS-validated module operation (Ubuntu FIPS kernels or equivalent). This is an accepted risk for the homelab boundary.
